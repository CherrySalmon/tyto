# AWS S3 Setup

Tyto stores student-submitted files in S3 in production. Uploads go direct from the browser to S3 via presigned POST; downloads go via a backend redirect to a presigned GET. The backend never proxies file bytes.

In `development` and `test`, the backend selects a local-filesystem `LocalGateway` instead — no S3 setup required. This document covers the production path only.

## What you need

- An AWS account with permission to create S3 buckets and IAM users.
- The production origin of your deployed app (e.g., `https://your-app.herokuapp.com`). Required for CORS.

## 1. Create the bucket

1. In the [S3 console](https://s3.console.aws.amazon.com/s3/), create a new bucket.
2. **Name**: globally unique across all AWS accounts. Suggested pattern: `tyto-<deployment>-uploads` (e.g., `tyto-prod-uploads`, `tyto-staging-uploads`) or `tyto-<environment>-uploads`.
3. **Region**: pick the region closest to your deploy. For Heroku's default `us-east-1`, choose `us-east-1`.
4. **Block Public Access**: leave **all four settings ON**. Presigned URLs do not require public access.
5. **Bucket Versioning**: leave **disabled**. Tyto uses deterministic keys with overwrite-on-resubmit; versioning would defeat that.
6. **Default encryption**: leave on (SSE-S3 / AES-256 is the AWS default for new buckets).

No bucket policy is needed — access is granted to a single IAM user via that user's policy.

## 2. Create the IAM user

The backend authenticates to S3 with a single IAM user's access keys. That user needs the minimum permissions to presign uploads/downloads and to HEAD/DELETE objects.

1. In the [IAM console](https://console.aws.amazon.com/iam/), create a new user with **programmatic access** only (no console login).
2. Attach a custom policy with the JSON below — replace `<your-bucket>` with your bucket name:

   ```json
   {
     "Version": "2012-10-17",
     "Statement": [
       {
         "Effect": "Allow",
         "Action": [
           "s3:PutObject",
           "s3:GetObject",
           "s3:DeleteObject"
         ],
         "Resource": "arn:aws:s3:::<your-bucket>/*"
       }
     ]
   }
   ```

3. Generate an **access key** for the user. Save the **Access Key ID** and **Secret Access Key** somewhere secure — you'll set them as environment variables in step 4.

Why these three actions:

- `s3:PutObject` — required for the IAM user whose credentials sign the upload POST policy. The signature delegates upload rights to the browser.
- `s3:GetObject` — covers both the `HEAD` performed by the backend during submission confirm and the presigned `GET` used for downloads.
- `s3:DeleteObject` — used when a student resubmits with a different file extension (best-effort cleanup of the old object).

## 3. Configure CORS on the bucket

The browser uploads directly to S3 (form-POST) and downloads via redirect (XHR-based GET). Both are cross-origin requests, so the bucket needs a CORS configuration.

In the bucket's **Permissions** tab, set the CORS configuration to:

```json
[
  {
    "AllowedHeaders": ["*"],
    "AllowedMethods": ["POST", "GET"],
    "AllowedOrigins": ["https://<your-app>.herokuapp.com"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3000
  }
]
```

Replace `<your-app>.herokuapp.com` with your production origin. If you deploy to a custom domain, list both the Heroku origin and the custom domain (the array can hold multiple origins).

## 4. Set production environment variables

The backend reads these at boot via `config/initializers/credentials.rb`. After the initializer runs, the values are removed from `ENV` and cached on `Tyto::FileStorage` — so consumer code never reads `ENV['S3_*']` directly.

```bash
heroku config:set S3_BUCKET=<your-bucket-name>
heroku config:set S3_REGION=<your-bucket-region>          # e.g., us-east-1
heroku config:set S3_ACCESS_KEY_ID=<iam-user-access-key>
heroku config:set S3_SECRET_ACCESS_KEY=<iam-user-secret>
```

`LOCAL_STORAGE_*` variables are not used in production and can be left unset.

## How dev vs prod selection works

`Tyto::FileStorage.build_gateway` consults `Tyto::Api.environment` (Roda's `:environments` plugin). The selection is an **allowlist**, not a denylist:

- `:development` and `:test` → `LocalGateway` (filesystem under `LOCAL_STORAGE_ROOT`)
- everything else (including `:production`, `:staging`, `:preview`) → `Gateway` (AWS S3)

The S3 Gateway raises `Tyto::FileStorage::ConfigurationError` at boot if any of the four `S3_*` variables is missing — a misconfigured deploy fails immediately rather than at first upload.

## Verifying the setup

After deploy, the smoke test in [doc/s3-smoke-test.md](s3-smoke-test.md) walks through one full upload → confirm → download cycle and explains how to read each failure mode (CORS rejection, IAM permission denial, region mismatch, signature errors).
